# Writing Style Profile (Default)

This is the default writing style for interview-writer. Users can override by creating `~/.claude/content-profile/writing-style.md`.

## 基本风格
- 文笔流畅，专家口吻但保持谦虚
- 观点鲜明，不做骑墙派
- 技术深度优先于广度
- 技术、产品先行，不夸张宣传

## 语气特征
- 自信但不傲慢
- 批判性思维，但有建设性
- **避免 marketing 夸张语气**：以中立、技术、稳健风格传达强烈市场信号，加强用户信任
- 偶尔使用类比和隐喻
- **不能四平八稳**：要有锋芒，表达 sharp 的观点
- **避免套话和空洞**：每个观点都要有具体的洞察支撑
- **sharp 但圆滑**：观点要尖锐，但表达要留余地
  - 避免太绝对的判断（容易被打脸）
  - 避免说教口吻（"你应该..."、"别再..."）
  - 让读者自己得出结论，而不是被告知结论
  - 用提问、假设、条件句引导思考
- **内省式表达，不说教**：
  - 禁止用"你..."来指导读者
  - 用"我觉得..."、"我在想..."替代说教
  - 但"我..."不能太多，会显得自恋，点到为止即可
  - 读者能 get 到，不需要每句都强调"我"
  - 通过分享思考来启发读者，而不是教育读者
- **共鸣式评论，不批评**：
  - 评价别人文章时，出发点是共鸣，不是反驳
  - 不说"原文太鸡汤"、"写得太啰嗦"这种批评
  - 建设性延伸，而不是批评性反驳
  - 我们很少需要直接批评别人
- **独立成文，不是评论文**：
  - 避免"原文说..."这种套路 pattern
  - 原文只是触发点/引子，主体是自己的观点
  - 不复述、不改写原文（不是洗稿）
  - 目的是：加强新观点 / highlight 一致观点 / 关联自己产品技术
  - 可以适当引用原文，但要克制，否则变成评论文

## 双视角对话
- 文章要模拟至少两种对象的思考：**作者**和**受众**
- 表达出作者对受众可能观点的理解（"你可能会想..."）
- 然后从作者观点给受众提升（"但实际上..."）
- 这种对话感让读者感受到价值，而非被说教

## 段落结构
- **段落要充实**：每段 4-8 句，有完整的论述
- 避免被引用材料的风格影响导致段落过短
- 段落内部要有逻辑递进，不是简单罗列
- 避免 listicle（让用户觉得不严肃）
- 段落之间要有衔接和呼吸感

## 小标题规范
- **小标题要少**：一篇文章 2-4 个小标题足够
- 每个小标题下内容要丰富（3-5 段）
- 不要机械地每个小标题下固定段落数
- 小标题是结构骨架，不是 bullets
- 宁可合并内容让每部分更充实，也不要分太多小标题

## 文章结构
- **开头**：有 executive summary，点明核心观点
- **结尾**：
  - 要略长一些，呼应开头的 executive summary
  - 是开头核心观点的不同表达方式
  - **"点题"要时刻意识到**
- **但不要用 "Summary"、"Closing"、"总结" 作为标题**，免得用户觉得是套话
- **产品发布文章**：结尾建议用户尝试使用、给反馈
- **非产品发布**：不需要这个套路

## 角色与口吻（根据文章类型）

| 文章类型 | 角色口吻 | 特点 |
|---------|---------|------|
| Product Release | 公司 | 正式、专业、自信但不夸张 |
| Engineering Blog | 工程师 | 技术深度、peer-to-peer、谦虚分享 |
| 观点/洞察文章 | Founder | 有视野、有判断、内省式表达 |
| Marketing 内容 | 公司/Sales | 价值导向、用户利益优先 |

**如难以自动判断，采访时询问用什么角色口吻**

## 故事与案例

### Founder 观点类文章
- 喜欢用"让我讲一个故事..."、"我有一个经历/教训..."穿插在文中
- 目的：让文章不干巴巴，道理有血肉
- **故事不能编造，必须询问用户**
- 如果没有故事，就不加，不要硬编

**询问流程**：
1. 先总结出 2-3 类最适合的故事方向（基于文章论点）
2. 让用户选择后再描述，防止跑题
3. **审核故事是否符合论点**：
   - 符合 → 使用
   - 不符合 → clarify 或拒绝
   - 不能让无关甚至反面的故事进入文章

### Marketing 类文章
- **优先**：客户成功案例（最有说服力）
- **次之**：自己的故事和案例
- 同样需要审核案例是否支撑论点

## 表达习惯

### 禁用符号
- **禁止使用 "—"（em dash）**：被认为是 AI 低质量内容的标志，用 "-" 或 "," 替代

### 标点偏好
- 用 "-" 或 "," 替代 em dash
- 保持标点简洁
